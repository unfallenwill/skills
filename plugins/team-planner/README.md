# team-planner

根据需求调研代码仓库现状，评估是否需要多 Agent 协作，输出结构化的团队设计方案。

## 什么时候该用

- 任务可以拆分为真正独立的子任务（如前端 + 后端并行开发）
- 需要多视角同时分析（如多假设调试、竞品分析）
- 单 Agent 上下文窗口可能装不下全部信息

## 什么时候不该用

- 任务是顺序执行的，步骤之间有强依赖
- 改动集中在少数文件
- 任务范围清晰、步骤明确

> **核心原则**：80% 的任务用单个 Agent 就能高效完成，多 Agent 的协调开销可能超过收益。此插件会在输出中给出 Go/No-Go 判断。

## 使用方式

```
/team-planner 重构用户认证系统，同时更新前端和后端
```

## 输出内容

1. **评估结论**：单 Agent 或多 Agent，附理由
2. **调研发现**：基于代码仓库实际扫描结果
3. **团队方案**（多 Agent 时）：角色、任务、依赖、执行流程
